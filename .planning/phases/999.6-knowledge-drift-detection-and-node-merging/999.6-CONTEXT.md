# Phase 999.6: Knowledge Drift Detection and Node Merging - Context

**Gathered:** 2026-06-19
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase delivers entity resolution and graph self-healing during ingestion in the Rust RAG engine. Instead of creating duplicate nodes or destructively mutating existing entities, the system identifies potential synonyms and links them non-destructively in the graph. v1 leaves ports by defining a clean asynchronous `EntityResolver` trait and implementing a pass-through resolver.

</domain>

<decisions>
## Implementation Decisions

### Data Mutation Policy
- **D-01:** Implement **Non-Destructive Linking**. To preserve source traceability and citation auditability, the system will not delete or overwrite duplicate entity nodes. Instead, it creates a `SAME_AS` or `SYNONYM_OF` relationship edge between duplicate nodes in the `edges` table.
- **D-02:** At query time, the Rust RAG traverser will follow these linking edges to aggregate synonym entity context during query resolution.

### Candidate Generation Strategy
- **D-03:** Use **Vector Search Similarity** for candidate generation. When a new entity is extracted, embed its name and query the LanceDB `nodes` table's `name_vector` to find the top-K similar entity nodes as candidate duplicates.

### Ingestion Integration Hook
- **D-04:** Define an asynchronous **`EntityResolver` trait** in Rust with a method:
  `async fn resolve(&self, extracted: Vec<RawNode>, store: &LanceStore) -> Result<Vec<ResolvedNode>>;`
- **D-05:** In v1, implement a pass-through `ExactMatchResolver` that writes nodes directly to the database. This allows the modular `SemanticLinkResolver` to be swapped in later in Phase 999.6.

### Agent Discretion
- The agent has discretion over the similarity thresholds for selecting candidate duplicates, the exact edge naming convention (e.g. `SAME_AS`), and the prompt structure for verifying synonym relationship links.

</decisions>
